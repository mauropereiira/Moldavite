Building Plugins

Add your own commands to Moldavite's command palette and slash menu with a few lines of JavaScript. This is the v1 plugin API.

Trust model — read this first

Plugins run in a sandboxed Web Worker with no DOM, no Zustand stores, no Tauri IPC, and no network globals (fetch, XMLHttpRequest, WebSocket, etc. are removed from the worker's global scope). The only channel back into the app is a postMessage bridge to the curated PluginAPI, and permissions are enforced host-side — a plugin can't reach a method its manifest didn't declare, even if it tries to bypass the API object it was handed. Consent is pinned to a SHA-256 hash of the plugin's code, so any change re-prompts you.

Still: enabling a plugin means running someone else's code. Only enable plugins you trust.

Plugin anatomy

A plugin is a folder inside your Forge's .plugins/ directory (your Forge is your notes directory, e.g. ~/Documents/Moldavite/Default/): 

<forge>/.plugins/my-plugin/
├── manifest.json     # identity + declared capabilities
├── plugin.js         # ES module; default-exports register(api)
└── README.md         # optional

To get started:

  1. Create the folder .plugins/<plugin-id>/ in your Forge.
  2. Add a manifest.json and a plugin.js (both below).
  3. Open Settings → Plugins, find your plugin, toggle it on, and accept the permission sheet.
  4. Run your command from the command palette (⌘ P) or the editor slash menu (type /).

Or click Settings → Plugins → Install example plugin to drop a working example (moldavite-example) into your Forge and read its source — it's the same one walked through at the end of this page.

manifest.json

{
  "id": "my-plugin",
  "name": "My Plugin",
  "version": "1.0.0",
  "author": "You",
  "description": "What it does.",
  "apiVersion": 1,
  "permissions": ["editor", "ui"]
}
FieldRequiredNotes
idyesMust equal the folder name; lowercase letters, digits, and hyphens (^[a-z0-9][a-z0-9-]*$).
nameyesDisplay name shown in the plugin manager and permission sheet.
versionyesSemver.
apiVersionyesMust be 1. Other values are shown as incompatible and not loaded.
authornoShown in the manager and permission sheet.

permissionsnoWhich API groups you use: "editor", "ui". Enforced at runtime — see below. commands is always available.

plugin.js — the entry point

plugin.js is an ES module whose default export is a register(api) function, called once when the plugin loads: 

export default function register(api) {
  api.commands.add({
    id: 'greet',
    label: 'Insert greeting',
    handler: async () => {
      await api.editor.insertText('Hello from my plugin!');
    },
  });
}

Every editor and ui method is async — they cross a postMessage bridge to the main thread. await them in your handlers (or fire-and-forget for calls whose result you don't need).

Plugins load on app start and after a Forge switch; there's no hot-reload in v1 — edit plugin.js, then reopen the app (the Plugins tab also re-scans). Plugins load over a dedicated plugin:// scheme rather than eval, and can only be served from inside the active Forge's .plugins/ directory.

The v1 API surface

interface PluginAPI {
  app: { version: string; apiVersion: number };

  // Always available.
  commands: {
    add(cmd: {
      id: string;      // namespaced to `${pluginId}:${id}` internally
      label: string;   // shown in palette + slash menu
      handler: () => void | Promise<void>;
    }): void;
  };

  // Requires "editor" in manifest permissions. All calls are async.
  editor: {
    getActiveNote(): Promise<{ title: string; content: string } | null>;  // content is HTML
    insertText(text: string): Promise<void>;                              // at the cursor
  };

  // Requires "ui" in manifest permissions.
  ui: {
    toast(message: string, kind?: 'info' | 'success' | 'error'): Promise<void>;
  };
}
  • Permissions are enforced. Calling api.editor.* or api.ui.* without declaring the matching permission throws (and Moldavite shows an error toast).
  • Errors are contained. If register or a command handler throws, Moldavite logs it ([plugin:<id>] in the console) and shows a toast — it never crashes the app.

The permissions model

When a user enables your plugin, Moldavite shows a permission sheet listing the plugin's name, author, description, and every permission it declares. The plugin only runs after the user accepts.

Since v1.5, that consent is pinned to a SHA-256 content hash of manifest.json + plugin.js. If anything in the plugin changes — a new version, an edited handler, a single character — the hash no longer matches and the user is re-prompted before the plugin runs again. A plugin can't silently swap out its code after being granted access.

  • Enable state is per-Forge — a Work Forge and a Personal Forge can run different plugins.
  • Uninstall (Settings → Plugins) deletes the folder from your Forge and forgets the grant.

How users run your commands

Every command you register appears in two places:

  • the command palette (⌘ P), under a "Plugins" group, and
  • the editor slash menu (type /), with a puzzle-piece icon.

Selecting one runs your handler.

Walkthrough: the example plugin

This is the full source of the bundled example plugin (installable from Settings → Plugins). It registers two commands and uses both gated permission groups (editor, ui) on top of the always-available commands API: 

// Moldavite example plugin — demonstrates the v1 command API.

export default function register(api) {
  // Insert the current time at the cursor.
  api.commands.add({
    id: 'insert-timestamp',
    label: 'Insert timestamp',
    handler: async () => {
      await api.editor.insertText(new Date().toISOString());
    },
  });

  // Toast the word count of the active note.
  api.commands.add({
    id: 'word-count',
    label: 'Word count',
    handler: async () => {
      const note = await api.editor.getActiveNote();
      if (!note) {
        await api.ui.toast('No active note', 'error');
        return;
      }
      const text = note.content.replace(/<[^>]+>/g, ' ');
      const words = text.trim().split(/\s+/).filter(Boolean).length;
      await api.ui.toast(`${words} word${words === 1 ? '' : 's'}`, 'info');
    },
  });
}

A few things worth noticing:

  • Command IDs are local to your plugin. insert-timestamp becomes my-plugin:insert-timestamp internally, so plugins can't collide.
  • getActiveNote() can return null — always handle the no-note case, like the word-count command does with an error toast.
  • Note content is HTML. The word counter strips tags with a regex before counting — do the same before any text processing.
  • Because this plugin calls api.editor.* and api.ui.*, its manifest must declare "permissions": ["editor", "ui"].
What's next for the API

On the roadmap: network fetch with a per-host allowlist, context-menu items, right-panel widgets, note read/write, ui.prompt, per-handler timeouts, and stricter isolation so the curated API becomes a true security boundary. Follow along in the GitHub repo.
